## Important (must-ship for v0.4)

### I-1: `--deep` mode — PDF body analysis for the 4-bot pipeline

Currently the 4-bot pipeline only sees `title + abstract + authors` per paper. Skeptic and Proposer hit a ceiling — abstracts skip method details, ablations, and limitations that are decisive for "is this gap real?" / "is this proposal novel?". Add `--deep` flag to `/find-topic` that fetches the PDF for the top-K matched papers and feeds extracted body text into Skeptic / Proposer as additional context.

Sub-tasks:
- **PDF fetch**: `Paper.pdf_url` is already populated for arXiv (canonical), OR-native (`openreview.net/pdf?id=…`), and dblp imports (varies — e.g., `ojs.aaai.org/…`, `ieeexplore.ieee.org/…`). Add a fetcher with primary-then-fallback (and graceful skip if 4xx/5xx).
- **Text extraction**: `pypdf` (lightweight) or `pdfminer.six` (better layout). Strip references / figure captions / appendix before LLM ingestion.
- **Token budget**: PDF body ≈ 8-15K tokens/paper. K=20 → 160-300K, exceeds Sonnet 200K. Either (a) cap K=10 with rerank-then-include, or (b) extract sections (intro / method / limitations) before passing — the latter aligns with what Skeptic/Proposer actually need.
- **Caching**: PDF text is immutable per paper id. Cache to `metadb/.pdfs/<paper_id>.txt`; sweep on prune so it tracks the rolling window.
- **Wiring**: only Skeptic + Proposer consume body text in v0.4; Trend / Gap stay on metadata-only (cheap, broad pattern matching).
- **Prior history**: deferred from v0.1 → v0.2 → v0.3. See `SKILL.md` §9 and design doc §5 (out-of-scope notes).

### I-2: Journal coverage (not just conferences) — **shipped**

Layer 1 currently scrapes arXiv (preprint) + HF (curation) + OR/dblp conferences. Journals (TPAMI, JMLR, TMLR, Nature Machine Intelligence, IEEE TNNLS, ACL/CL Transactions) are untouched. For applied/theoretical and longer-form research questions, journal papers carry weight that conferences don't.

**Shipped (2026-04-28):** `collector/src/scrapers/journal.py` queries OpenAlex
by ISSN. Initial `JOURNAL_TARGETS` configured for IEEE Trans. Industrial
Informatics (1551-3203) and Expert Systems with Applications (0957-4174).
`--with-journal` opt-in mirrors `--with-or` / `--with-s2`; backfill wires it
as one-shot. Empty-abstract papers (TOCs/errata) dropped at scraper level;
~85 valid papers per venue per pull (~57% of raw OpenAlex results). Journal
venues weight=3 in `match_substring._VENUE_WEIGHT` (= AAAI tier), placing
them above arXiv preprints but below top-tier conferences. Add new journals
by appending to `JOURNAL_TARGETS` + adding a weight entry.

Sub-tasks:
- **Source pick**: candidates —
  - **Crossref REST** (`https://api.crossref.org/works`) — free, no key, ISSN-filterable. Broadest journal coverage.
  - **OpenAlex** (`https://api.openalex.org/works`) — free, structured `host_venue`/`type` fields, citation graph.
  - **Semantic Scholar** — already integrated; filter via `publicationTypes:JournalArticle`. Same API-key constraint as conferences.
  - **dblp** — has journal listings (`db/journals/<acronym>`); aligns with our existing OR/dblp pattern.
  - Recommend OpenAlex or Crossref (no API-key barrier; rich metadata).
- **Scraper module**: `collector/src/scrapers/journal.py` parallel to others, gated behind `--with-journal` (opt-in like OR/S2 in v0.3).
- **Config**: `JOURNAL_TARGETS` list (ISSNs or OpenAlex venue ids) in `config.py`. Start with ~6-8 high-signal journals, expand as needed.
- **DB**: reuse `Paper` model; `venue` = journal name; `source = "journal"` (or per-API: `"openalex"` / `"crossref"`). `also_in` merge handles dedup with arXiv preprints.
- **Volume control**: TPAMI alone is ~2000/year. Per-journal cap + 500-day window prune handles spike. Decide if `categories` field needs new buckets for non-cs journals.
- **Backfill semantics**: like OR/S2, journal scrapers usually ignore `target_date` (return recent N) — wire as **one-shot** in `backfill.py` alongside OR/S2.

### I-3: Gemini API for the 4-bot pipeline (token-cost reduction) — **shipped**

Sonnet 4.6 calls × 4 bots add up — each `/find-topic` run is on the order of $0.40-$0.60 uncached, ~$0.20-$0.30 with prompt caching. Gemini 2.5 Pro / Flash offer comparable reasoning at materially lower cost (Flash especially), and Google's API has its own prompt-caching primitive (`cached_content`). Add a model selector that routes the 4-bot pipeline through Gemini.

**Shipped (2026-04-28):** `run_pipeline.py` dispatches between Anthropic
(`cache_control: ephemeral`) and Google (`cached_content`) backends per
`--model {sonnet|gemini-pro|gemini-flash}`. **Default flipped to
`gemini-flash`** (~$0.01/run vs Sonnet ~$0.20). Per-bot token telemetry +
USD estimate written to `usage.json`. Mixed-model (Flash for Trend/Gap,
Pro/Sonnet for Skeptic/Proposer) deferred to v0.5 — single-model is good
enough on the empirical runs to date.

Sub-tasks:
- **CLI surface**: `/find-topic ... --model {sonnet|gemini-pro|gemini-flash}`; default stays `sonnet` until Gemini path is regression-tested.
- **SDK**: `google-genai` (https://pypi.org/project/google-genai/). Key from `GOOGLE_API_KEY` env (add to `.env.example`).
- **Prompt caching parity**: replicate the I-3 (v0.3) pattern — matched-papers JSON block as `cached_content` so Trend's call seeds the cache and Gap/Skeptic/Proposer reuse it.
- **JSON-mode parity**: each bot's prompt is currently shaped for Sonnet's JSON-output behavior. Gemini's `response_schema` is stricter; either (a) port prompts to use `response_schema`, or (b) keep free-form and post-parse with the existing fault-tolerant JSON loader.
- **Quality gate**: don't promote to default until 5 demo runs reproduce the Sonnet output within tolerance (±10% on cluster count / proposal count, no missing required fields).
- **Cost telemetry**: extend `_run_pipeline.py` to log per-bot input+output tokens and a $-equivalent line so the savings are visible in CLI output.
- **Mixed-model option (stretch)**: route Trend + Gap to Gemini-Flash (cheap pattern-finding) while Skeptic + Proposer stay on Sonnet (sharper reasoning). Saves 50-70% even without full migration.

### I-4: Flash truncation safety net — **shipped**

Earlier observation: `gemini-flash` would truncate Trend output mid-array on
K=3 with 100 matches even at `max_output_tokens=32768`. Two changes applied:
(a) `trend_analyzer.md` caps `paper_ids` at 30 per cluster so output stays
under ~2K tokens; (b) `call_gemini_sdk` now logs `finish_reason` per call so
any future MAX_TOKENS hit surfaces immediately rather than silently producing
malformed JSON. Re-running `llm unlearning` (K=3, max-papers=100) post-fix:
all 4 bots STOP at 1.5–1.8K out tokens. Mark closed; if it recurs the
log line is the breadcrumb.
