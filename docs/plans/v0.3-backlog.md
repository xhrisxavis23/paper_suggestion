# v0.3 Backlog

Findings from v0.2 implementation (`9f8e842`) + post-v0.2 monthly-DB refactor (master `71207fa`).

## From v0.2 review

### Important (correctness / scaling / cost)

- [ ] **I-1**: **Token budget enforcement.** No hard cap on prompt size for the 4-bot pipeline. Empirically, a 60-day window gives **2,545 matches for "llm safety"**, **6,563 for "rag"**, **3,511 for "agent"** — at 1500 chars/abstract that's 900K–2.3M tokens, well past Sonnet's 200K context. The existing `_run_pipeline.py` masks the issue with a manual `max_papers=80` cap, but `SKILL.md` §4 has no such cap and would silently OOM. Add: (a) hard cap (e.g., 200 papers / 100K input tokens) with deterministic ranking (date-desc + venue weight) before passing to Trend-Analyzer, (b) `--max-papers N` CLI flag, (c) warning at scope-confirmation when matches > cap.

- [ ] **I-2**: **Backfill CLI.** Right now backfill is a gitignored ad-hoc shell script (`_backfill.sh`). A fresh clone has no DB until cron runs for 60 days. Add `python -m collector.backfill --start YYYY-MM-DD --end YYYY-MM-DD [--sources arxiv,hf]` that iterates day-by-day, respects `--skip-*` flags, and prints final coverage stats. Document in README installation section.

- [ ] **I-3**: **Prompt caching for the 4-bot pipeline.** Trend-Analyzer / Gap-Hunter / Skeptic / Proposer all receive the same matched-papers JSON block (~80 papers × 1.5K chars ≈ 30K tokens of static input). With Anthropic's `cache_control` this can be cached once and re-used 3× — typical ~50% cost cut on the same pipeline. Update `skills/topic_finder/SKILL.md` and the driver script to mark the matched-papers section as `ephemeral` cached.

- [ ] **I-4**: **Stats history (append-only).** `metadb/stats.json` is overwritten every run. If S2 starts rate-limiting on day 12 of a 30-day issue, you can't tell from the file alone. Add `metadb/stats_history.jsonl` (one row per run); keep `stats.json` as the latest-snapshot for compat. CI prunes history rows older than 90 days.

- [ ] **I-5**: **OR / S2 are de facto dead in the rolling window.**
  - **OpenReview**: only `ICLR.cc/2026/Conference` configured; all 1000 returned submissions are dated 2025-09 → 100% pruned by 60-day cutoff. Either rotate `OPENREVIEW_VENUE_IDS` to active venues with `cdate` in-window, or drop OR from the default sources and document.
  - **Semantic Scholar**: without `SEMANTIC_SCHOLAR_API_KEY`, every venue rate-limits to 0 (now visible in `stats.json.failures`). Either make the key required at startup (loud failure) or remove S2 from the default scraper list. Current state silently contributes nothing.

### Minor

- [ ] **M-1**: **arXiv weekend digest.** Sat/Sun runs produce "총 수집: 0건" digest because arXiv has no weekend submissions. The `_date_window` weekend rollback handles fetching, but the digest still shows 0 because nothing *new* was added. Either skip the digest write on weekends or merge weekend fetches into Monday's digest.

- [ ] **M-2**: **Embedding-based matching (opt-in).** Long-deferred (v0.1 → v0.2 → v0.3). Substring-only matching misses semantically near papers (e.g., "ICL" vs "in-context learning" caught, "few-shot" missed unless added to keyword list). Add `--match-mode embedding` using `sentence-transformers/all-MiniLM-L6-v2` + FAISS over the rolling DB. Index built lazily, cached at `metadb/.embeddings/`.

- [ ] **M-3**: **Tests for scraper failure-surfacing path.** I-4 added `self.failures` to each scraper, but no unit test asserts that a raised exception in `_run_source` populates the `stats.json.failures` list. Add a fault-injection test using `MagicMock` that throws on `fetch()`.

- [ ] **M-4**: **Tests for `_short_id` non-arxiv collision-resistance.** v0.2 M-2 added the `P-<VENUE>-<6hex>` form but no test covers it. Add an assertion that two papers with same long-prefix titles produce different short ids.

- [ ] **M-5**: **Auto-prune `reports/.cache/`.** Each `/find-topic` run drops a cache dir; on a busy machine this accumulates indefinitely. Add a sweep step (oldest > 30 runs deleted) at the start of each pipeline run.

- [ ] **M-6**: **Concurrent scraper fetches.** `arxiv` waits 3s × 6 categories = 18s pacing per run; `s2` waits 1s × 6 venues. Total wall time ~30s for fetches that could be parallelized inside each scraper (per arXiv's 1-req-per-3s rule per IP, that's still serial; but cross-source parallelism — arxiv + HF + OR + S2 in parallel — saves the full sum to max).

- [ ] **M-7**: **Run pytest in CI** (`.github/workflows/daily_collect.yml`) before the commit step. Currently a broken `db.py` would still push corrupted data.

- [ ] **M-8**: **`Paper.year` redundant.** Every `Paper` has both `year` and `published_date.year`. Drop the field, derive on access (`@property` or in serializers). Saves a few lines per JSONL row.

- [ ] **M-9**: **`.env.example`** at repo root listing `SEMANTIC_SCHOLAR_API_KEY=` to flag setup steps for new contributors.

- [ ] **M-10**: **HF / arxiv dedup loses metadata.** When an HF entry shares an arxiv id with an existing arxiv row, `RollingDB.append` skips the HF row entirely. The HF entry's `venue: "HF"` (signal that humans curated it) is lost. Decide policy: (a) keep arxiv row + add `also_in: ["hf"]` field, (b) prefer HF row's metadata on collision.

- [ ] **M-11**: **Report bibliography too flat.** Section 4 of the report lists 80+ papers in a single dump. Better UX: group by cluster name, link cluster anchors back to bibliography section, deduplicate the papers cited only in proposals vs. cluster top3.

- [ ] **M-12**: **Concurrent `append` safety.** `RollingDB.append` reads → filters → opens for append. Two simultaneous workflows (e.g., manual run + cron) could double-write a paper across files. Add an `flock`-style file lock or document that only one writer is supported.
